Executar com .NET Aspire

Recomendado para desenvolvimento

.NET Aspire é a maneira mais fácil de executar o projeto. Ele automaticamente:

  • Inicia containers Docker para PostgreSQL e Redis
  • Configura connection strings automaticamente
  • Executa a API e o Blazor UI em paralelo
  • Habilita OpenTelemetry com exportação OTLP
  • Fornece um dashboard de monitoramento

Passo a Passo

1

Clonar o repositório

git clone https://github.com/fullstackhero/dotnet-starter-kit.git
cd dotnet-starter-kit
2

Restaurar dependências

dotnet restore src/FSH.Framework.slnx
3

Certificar que Docker está rodando

docker ps # Deve retornar sem erro
4

Executar o AppHost

dotnet run --project src/Playground/FSH.Playground.AppHost

Aguarde até ver mensagens indicando que todos os serviços estão rodando

5

Acessar os endpoints

API
https://localhost:5285
Blazor UI
https://localhost:5286
Aspire Dashboard
https://localhost:17XXX
Hangfire
https://localhost:5285/jobs

Aspire Dashboard

O dashboard do Aspire mostra logs, traces, métricas e health checks de todos os serviços em tempo real. É extremamente útil para debugging!

Executar API Standalone

Sem Aspire (Docker/Postgres manual)

Passo a Passo

1

Iniciar PostgreSQL

docker run -d --name postgres \
-e POSTGRES_PASSWORD=postgres \
-e POSTGRES_DB=fsh \
-p 5432:5432 \
postgres:16-alpine
2

Iniciar Redis

docker run -d --name redis \
-p 6379:6379 \
redis:alpine
3

Configurar appsettings.Development.json

{
  "DatabaseOptions": {
    "Provider": "postgres",
    "ConnectionString": "Host=localhost;Port=5432;Database=fsh;Username=postgres;Password=postgres"
  },
  "CachingOptions": {
    "Redis": "localhost:6379"
  },
  "JwtOptions": {
    "SigningKey": "your-development-key-min-256-bits"
  }
}
4

Executar a API

dotnet run --project src/Playground/Playground.Api

API disponível em: https://localhost:7030

Executar Testes

Unit tests, integration tests e architecture tests

Todos os Testes

dotnet test src/FSH.Framework.slnx

dotnet test src/Tests/Identity.Tests

Com Cobertura

dotnet test --collect:"XPlat Code Coverage"

Teste Específico

dotnet test --filter "FullyQualifiedName~TestName"

Testa handlers, validators, domain logic isoladamente

Integration Tests

Testa endpoints completos com banco de dados em memória

Build & Publish

Compilar e publicar para produção

Build (Debug)

dotnet build src/FSH.Framework.slnx

Build (Release)

dotnet build -c Release src/FSH.Framework.slnx

Publish (Ready for deployment)

dotnet publish src/Playground/Playground.Api -c Release -o ./publish

Arquivos publicados estarão em ./publish

Validações de Produção

O Program.cs valida configurações críticas em produção:

  • • DatabaseOptions:ConnectionString não pode estar vazio
  • • CachingOptions:Redis deve estar configurado
  • • JwtOptions:SigningKey é obrigatório

Docker

Containerização da aplicação

Build da Imagem

# Na raiz do projeto
docker build -t fsh-api:latest -f src/Playground/Playground.Api/Dockerfile .

Executar Container

docker run -d -p 8080:8080 \
-e DatabaseOptions__ConnectionString="Host=host.docker.internal;..." \
-e CachingOptions__Redis="host.docker.internal:6379" \
-e JwtOptions__SigningKey="your-key" \
fsh-api:latest

Docker Compose (Stack Completa)

Você pode criar um docker-compose.yml com API + Postgres + Redis:

version: '3.8'

services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: fsh
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

  redis:
    image: redis:alpine
    ports:
      - "6379:6379"

  api:
    build: 
      context: .
      dockerfile: src/Playground/Playground.Api/Dockerfile
    ports:
      - "8080:8080"
    environment:
      - DatabaseOptions__ConnectionString=Host=postgres;Database=fsh;...
      - CachingOptions__Redis=redis:6379
      - JwtOptions__SigningKey=your-key
    depends_on:
      - postgres
      - redis

volumes:
  postgres_data:

Deploy (Terraform)

Infrastructure as Code para AWS

O projeto inclui módulos Terraform para deploy na AWS (ECS, RDS Postgres, ElastiCache Redis, ALB).

Estrutura Terraform

terraform/
├── modules/                    # Módulos reutilizáveis
│   ├── network/               # VPC, subnets, security groups
│   ├── ecs_cluster/           # ECS cluster
│   ├── ecs_service/           # ECS service + task definition
│   ├── rds_postgres/          # RDS PostgreSQL
│   ├── elasticache_redis/     # ElastiCache Redis
│   └── alb/                   # Application Load Balancer
│
├── apps/playground/           # Deployment do Playground
│   └── envs/
│       ├── dev/us-east-1/
│       ├── staging/us-east-1/
│       └── prod/us-east-1/
│
└── bootstrap/                 # Setup inicial AWS (S3 backend)

Deploy para AWS

1. Bootstrap (primeira vez)

cd terraform/bootstrap
terraform init
terraform apply

2. Deploy ambiente (dev/staging/prod)

cd terraform/apps/playground/envs/dev/us-east-1
terraform init
terraform plan
terraform apply

O que é Criado

  • • VPC com subnets públicas e privadas
  • • RDS PostgreSQL (Multi-AZ em produção)
  • • ElastiCache Redis cluster
  • • ECS Fargate cluster + service
  • • Application Load Balancer com HTTPS
  • • ECR repository para imagens Docker
  • • CloudWatch logs & metrics

Troubleshooting

❌ Erro: "Unable to connect to database"

Solução: Verifique se PostgreSQL está rodando e a connection string está correta

docker ps | grep postgres

❌ Erro: "Redis connection failed"

Solução: Inicie Redis ou desabilite caching temporariamente

builder.AddHeroPlatform(o => o.EnableCaching = false);

❌ Erro: "JWT validation failed"

Solução: Configure JwtOptions:SigningKey no appsettings

❌ Migrations não aplicam

Solução: Delete o database e reinicie, ou execute migrations manualmente

dotnet ef database update --project src/Playground/Migrations.PostgreSQL
